# Flow of System Call
## 서론
중간고사 기간이랑 겹치는 바람에 블로그 개설 3주만에 첫 글을 쓴다.  
운영체제에서 다루었던 시스템콜에 대해 정확하게 기억하기 위해 정리하고자 한다.  
이 글은 Monolothic Kernel을 기준으로 한다.

## 본론1: flow
### 1. 사용자 프로그램이 트랩을 발생
1. 시스템콜 번호를 eax 레지스터에 저장
2. `trap` 명령 실행 (주체: CPU)
    - PC, 상태 레지스터(flag) 등 최소한의 사용자 레지스터를 커널 스택에 저장
    - CPU 모드 전환(user mode -> kernel mode)

### 2. 트랩테이블을 참조하여 적절한 엔트리포인트로 점프

### 3. 시스템콜인터페이스 동작 (주체: 트랩핸들러(=시스템콜핸들러))
1. 아까 저장한 정보 외에 나머지 범용 레지스터 정보를 커널 스택에 저장
2. eax 레지스터에서 시스템콜번호 확인
3. 시스템콜테이블을 참조하여 해당 번호에 매핑되는 실제 시스템콜 메서드 호출

### 4. 호출된 시스템콜 실행

### 5. 종료 후 값 반환
1. `return-from-trap` 명령 실행
2. 결과를 eax 레지스터에 저장
3. CPU모드 전환(kernel mode -> user mode)
4. `trap` 다음 줄부터 계속 진행

## 본론2: Q&A
- Q. 1-1에서 시스템콜 번호는 어떻게 알고 있는가
    - 특정 파일에 컴파일 시점에 미리 매핑됨. ex) 리눅스의 `unistd.h`
- Q. 트랩테이블과 시스템콜테이블의 차이
    - 트랩테이블: 운영체제 부팅 시에 하드웨어에 등록되는 최상위 레벨의 표. 시스템콜 외에도 Timer Interrupt, Divide by zero 등 모든 종류의 예외 상황과 핸들러 주소 매핑돼있음.
    - 시스템콜테이블: 커널 내부에서 관리하는 시스템콜번호와 실제 메서드간의 매핑 표.
- Q. 시스템콜인터페이스란?
    - 핸들러가 번호 확인 후 실제 메서드로 연결해주는 전체적인 체계
- Q. 시스템콜은 의도된 트랩인가? 예외는 없나?
    - 시스템콜은 예외없이 무조건 의도된 트랩임.
- Q. 사용자 레지스터와 커널 레지스터의 차이
    - 물리적으론 같은 레지스터이고 사용 시점의 차이임.

## 본론3: Context Switching이 필요할 때 flow
3-3까지 동일.  
4번 진행중에 context switching 발생..
1. `swtch` 명령 실행
2. A프로세스의 커널 레지스터들을 A의 PCB에 저장
3. B의 PCB에서 레지스터 값들을 CPU로 읽어옴
4. **%esp를 B의 커널 스택으로 변경** -> 흔히 말하는 Context Switching 발생ㅇ
5. OS가 MMU의 base 레지스터 갱신

## 마무리
시스템콜은 사용자가 안전하게 하드웨어에 접근하기 위한 장치이다.  
과제로 xv6환경에서 시스템콜 제작했던 경험도 기회가 되면 정리해봐야겠다.
